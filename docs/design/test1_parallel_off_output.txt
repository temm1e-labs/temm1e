1) Rust ownership
- Core idea: every value has a single “owner” (a variable/binding) responsible for freeing it when it goes out of scope (RAII). This enables memory safety without a garbage collector.
- Moves vs copies: assigning/passing most types transfers ownership (a move), making the old binding unusable; types that implement Copy (e.g., integers) are copied instead.
- Borrowing: you can temporarily access a value without taking ownership via references: shared borrows `&T` (many allowed) or mutable borrows `&mut T` (only one at a time, and no shared borrows simultaneously). These rules prevent data races and use-after-free.
- Lifetimes: the compiler tracks how long references are valid; explicit lifetime annotations are sometimes needed to relate input/output reference lifetimes in function signatures.
- Common tools: `clone()` to duplicate owned data; smart pointers like `Box<T>`, `Rc<T>` (shared ownership single-thread), `Arc<T>` (shared ownership multi-thread), and interior mutability patterns (`RefCell`, `Mutex`) when mutation is needed under shared ownership.

2) async/await patterns
- Model: `async fn` returns a Future; `await` yields control until the future is ready. Futures are lazy—nothing runs until polled by an async runtime (Tokio, async-std).
- Concurrency patterns:
  - Run tasks concurrently: `join!`/`try_join!` (wait for multiple futures), `select!` (race/first-complete), `spawn` (background task).
  - Structured concurrency: prefer keeping spawned tasks scoped/owned so they’re awaited or cancelled deterministically; avoid “fire-and-forget” unless you handle shutdown.
  - Timeouts/cancellation: wrap with `timeout`, use cancellation tokens, and design loops to be cancellation-safe.
- I/O and CPU separation: use async for I/O-bound work; offload CPU-heavy work to blocking pools (`spawn_blocking`) to avoid starving the executor.
- Shared state: prefer message passing (channels) or `Arc<Mutex<_>>`/`RwLock` carefully; avoid holding locks across `.await` to prevent deadlocks and contention.
- Error propagation: use `?` in async functions the same way as sync; `try_join!` is useful when you want “fail fast” across concurrent operations.

3) Error handling with Result types
- `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`), making errors explicit and type-checked.
- Propagation: the `?` operator returns early on `Err`, converting error types via `From`/`Into` when possible.
- Designing error types:
  - Libraries often define a custom `enum` error type (possibly with `thiserror`) to preserve structured causes.
  - Applications sometimes use `anyhow::Error` for ergonomic, dynamic error aggregation with context.
- Adding context: attach information at the failure site (e.g., “failed to open config file”) so callers get actionable messages; use `map_err`, `with_context` (anyhow), or custom variants.
- Handling vs bubbling: handle errors where you can recover (retry, fallback, user prompt); otherwise bubble up with `Result` and let higher layers decide.
- Combinators: `map`, `and_then`, `or_else`, `unwrap_or`, `ok()` help transform/compose results; `match` is best when branching on specific error kinds.
- Best practices: avoid `unwrap()` in production paths; prefer returning `Result` from `main` (e.g., `fn main() -> Result<(), E>`) and centralize top-level reporting/logging.

## Threads
- **What it is:** OS-managed execution units running in parallel on one or more CPU cores.
- **How you structure work:** Spawn threads; share memory between them; coordinate with locks/atomics/condition variables.
- **Strengths:** True parallelism for CPU-bound work; straightforward mental model for blocking I/O; mature tooling.
- **Costs/risks:** Higher overhead per thread (stack, scheduling); contention and deadlocks; data races if synchronization is wrong.
- **Best fit:** CPU-heavy workloads, legacy blocking libraries, low-level systems where you need explicit control.

## Async/await
- **What it is:** Cooperative concurrency: many tasks share a small number of OS threads; tasks yield at `await` points while waiting (usually on I/O).
- **How you structure work:** Write `async` functions returning futures/promises; an event loop/runtime polls tasks; use combinators like join/select.
- **Strengths:** Scales well for high-concurrency I/O (many sockets/requests) with low overhead; avoids one-thread-per-connection.
- **Costs/risks:** Not automatic parallelism for CPU work (needs worker threads); complexity around cancellation, backpressure, and not blocking the runtime; debugging can be harder.
- **Best fit:** Network servers, clients, pipelines with lots of waiting on I/O.

## Actors
- **What it is:** Concurrency via isolated state + message passing. Each actor processes messages sequentially, typically from a mailbox.
- **How you structure work:** Model components as actors; communicate by sending messages; supervision/restarts often built in.
- **Strengths:** Reduces shared-memory bugs (no direct shared mutable state); good fault isolation; naturally distributed-friendly.
- **Costs/risks:** Message design and ordering semantics can be tricky; latency/throughput depend on mailbox/backpressure; cross-actor transactions are harder.
- **Best fit:** Systems with many independent entities (sessions, devices), distributed services, and applications needing resilience via supervision.
