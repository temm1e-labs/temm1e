1) Rust ownership
- Core idea: every value has a single “owner” (a variable/binding) responsible for freeing it when it goes out of scope. This enables memory safety without a garbage collector.
- Moves vs copies: assigning/passing most values transfers ownership (“move”), making the old binding unusable. Types that implement Copy (e.g., integers) are copied instead.
- Borrowing: you can lend access without transferring ownership via references: 
  - shared borrow: &T (many allowed at once)
  - mutable borrow: &mut T (only one at a time)
- Borrow checker rules (simplified): either many readers or one writer; references must not outlive the data they point to. Lifetimes express/verify this.
- Practical impact: APIs often take &T for read-only, &mut T for in-place mutation, or T when they need to consume/own the value.

2) async/await patterns
- What it is: async/await in Rust builds state machines (Futures). A Future is polled by an async runtime (Tokio, async-std, etc.) until it’s ready.
- Common patterns:
  - Concurrency with join: run independent tasks concurrently and await all results (e.g., join!/try_join!).
  - Spawning tasks: move work onto the runtime scheduler (tokio::spawn) for background/concurrent execution; requires Send + 'static in many cases.
  - Timeouts/cancellation: wrap futures with timeout; cancellation is typically cooperative—dropping a future cancels it if it’s not otherwise running.
  - Streams: for sequences of async values, use Stream (like async iterators) and combinators (buffered, map, filter).
  - Avoid blocking: don’t call blocking I/O or heavy CPU work on async executor threads; use spawn_blocking or a dedicated thread pool.
- Pitfalls:
  - Holding locks across .await can cause deadlocks/contention.
  - Excessive task spawning can add overhead; prefer structured concurrency (join) when possible.
  - Mixing runtimes or using non-async libraries can complicate design.

3) Error handling with Result types
- Result<T, E>: idiomatic Rust error handling returns either Ok(T) or Err(E). This makes failure explicit and composable.
- Propagation: the ? operator returns early on Err, greatly reducing boilerplate. Works with functions returning Result (and some other Try types).
- Designing error types:
  - Use specific enums for library/domain errors (e.g., enum MyError { Io(io::Error), Parse(ParseIntError), ... }).
  - Implement/derive std::error::Error and Display for good messages.
  - Use thiserror for ergonomic custom errors; use anyhow for application-level “catch-all” errors with context.
- Adding context: attach additional information when propagating (anyhow::Context or custom variants) so failures are diagnosable.
- Conversions: implement From to allow ? to automatically convert underlying errors into your error type.
- When to use panic!: reserve for programmer bugs/invariants, not expected runtime failures; prefer Result for recoverable errors.

### Threads
- **Model:** OS-managed threads run code in parallel; the scheduler time-slices or runs them on multiple CPU cores.
- **How you structure work:** Split work into functions/tasks and run them on separate threads; coordinate with shared memory.
- **Coordination primitives:** Mutex/RwLock, condition variables, atomics, semaphores, channels.
- **Strengths:** True parallelism for CPU-bound work; straightforward mental model for “do these things at the same time”; works well with blocking I/O.
- **Weaknesses:** Higher overhead per thread (stack, context switches); shared-state bugs (races, deadlocks); scaling to very high concurrency can be costly.
- **Best fit:** CPU-heavy workloads, mixed workloads with blocking libraries, simpler services where thread count is modest.

### Async/await
- **Model:** Cooperative concurrency using an event loop/runtime; tasks (futures) yield at `await` points so one thread can drive many tasks.
- **How you structure work:** Write async functions; compose tasks with `await`, `join`, timeouts; runtime polls tasks when I/O is ready.
- **Coordination primitives:** Async-aware mutexes/locks, channels, select/join combinators, cancellation via dropping/abort.
- **Strengths:** Excellent for I/O-bound workloads with many concurrent connections; low overhead per task; avoids one-thread-per-connection.
- **Weaknesses:** Not automatically parallel for CPU work (needs thread pool/spawn_blocking); “don’t block the executor” constraint; can be tricky with lifetimes/borrows (in Rust) and with holding locks across `await`.
- **Best fit:** High-concurrency network servers/clients, pipelines dominated by waiting on I/O.

### Actors
- **Model:** System is composed of actors that own their state and communicate by sending messages; each actor processes messages sequentially.
- **How you structure work:** Define actor types with mailboxes; send messages/commands/events; actors may spawn other actors.
- **Coordination primitives:** Message passing (mailboxes), supervision trees, routing, backpressure via mailbox limits.
- **Strengths:** Reduces shared-memory contention by isolating state; easier reasoning about state changes (single-threaded per actor); good fault isolation and resilience patterns (supervision/restarts).
- **Weaknesses:** Message design and serialization overhead; debugging distributed message flows can be harder; can introduce latency if mailboxes back up; not ideal for fine-grained shared data.
- **Best fit:** Systems with many independent stateful components (chat, games, IoT), distributed services, fault-tolerant architectures.
